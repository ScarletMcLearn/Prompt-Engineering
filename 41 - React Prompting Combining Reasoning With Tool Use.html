ReAct: Using External Tools Together with the Model

No matter how powerful Large Language Models are, they cannot do everything entirely on their own. To be truly effective, they often need access to external data sources and additional tools. Sometimes a model must retrieve up-to-date information, perform exact calculations, or use a specialized service — and that goes beyond its core capabilities.

So the question is:

How do you write a prompt so the model can “call” external tools, retrieve results from them, and then use those results in its reasoning?

One approach is called ReAct (Reason + Act). The idea is to teach the model to:

  • Plan a sequence of solution steps

  • Recognize when external information is needed

  • Choose the appropriate tool

  • Retrieve a result from the tool

  • Return to reasoning using that result

This approach is similar to chain-of-thought prompting, but expands it by adding actions — tool usage.

Walkthrough Example (Training Scenario)

The next section contains a long example. First, we demonstrate the pattern on one scenario, and then apply the same pattern to similar tasks.

Important: The examples below are demonstrations of the ReAct format (“Task / Think / Action / Result”).
The “Result” lines represent data retrieved from tools in the scenario, not something the model should invent.

Training Example: Planning Arrival Time for an Event

Task:
Calculate what time we should arrive at a BMX event so that my child makes their race slot (9:00–10:00 open race).

BMX is a cycling discipline. In competitions like this, there are many short heats (“motos”), and each one lasts about 30 seconds.

To determine arrival time, we need to understand:

  • When the races start

  • How many heats happen before the target category

  • How long each heat takes

Task:
Calculate when we should arrive at the BMX event so my child makes the 9:00–10:00 slot.

Think:
I need to find what time the first heat begins. I can look up the official schedule.

Action:
search official event page schedule

Result:
All heats start at 09:00.

Think:
I need to find how many heats happen before the 9:00–10:00 open race category. I can look it up in the race order.

Action:
search official event page race order

Result:
28 motos occur before this category.

Think:
I need to estimate how long one moto takes. I can use a video archive and time the first 10 motos.

Action:
open video archive and time first 10 motos

Result:
The first 10 motos took 6 minutes.

Think:
If 10 motos took 6 minutes, then 28 motos will take about 16.8 minutes (round to 17 minutes).
If the event starts at 09:00, the target category should begin around 09:17.
To be safe, arrive 30 minutes earlier — around 08:45.

Notice what this demonstration teaches the model:

  • How to reason step by step

  • When to use a tool

  • How to insert tool output

  • How to continue calculations afterward

Example 2: Estimating When an Oral Exam Ends

Task:
Determine when an oral exam will end if I am 15th in the list.

Think:
I need to know what time the exam starts.

Action:
search “oral exam schedule faculty of economics”

Result:
The exam starts at 10:00.

Think:
I need to know how much time is allocated per student.

Action:
search “oral exam rules faculty of economics”

Result:
Each student gets 12 minutes.

Think:
There are 14 students before me.
14 × 12 = 168 minutes.
168 minutes = 2 hours 48 minutes.
If the start time is 10:00, my turn begins around 12:48.
Add 12 minutes for my exam → finish around 13:00.

Example 3: Estimating Train Arrival Time

Task:
Determine when a train will arrive if the distance and average speed are known.

Think:
I need to find the distance between the cities.

Action:
search “distance between the two cities”

Result:
Distance is 400 km.

Think:
I need the train’s average speed.

Action:
search “average speed of the train on this route”

Result:
Average speed is 160 km/h.

Think:
Travel time = 400 / 160 = 2.5 hours.
If departure is at 14:00, arrival will be around 16:30.

How This Works Technically

In a real system, the process can work like this:

  • The model reaches the Action line

  • Execution pauses

  • The system calls a real tool (search, API, video, etc.)

  • The tool output is inserted into the Result line

  • The updated context is sent back to the model

  • The process continues until the task is complete

This way, the model does not “make up” data — it obtains it from external sources.

Key Observations

  • The model learns tool usage from few-shot examples

  • It adopts the tool-calling format

  • It can recognize URL-like patterns

  • It learns when a search is needed

  • It combines reasoning and action

An alternative approach is also possible: instead of teaching through examples, you can describe the available tools and the format for using them. Then the model can choose the right tool based on that specification.

Connection to Frameworks
