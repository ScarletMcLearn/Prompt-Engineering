Common Mistakes in Few-Shot Prompts

Let’s examine several examples where few-shot prompts fail.
In other words: what can go wrong in formatting, why the model “drifts” from the task, and what conclusions we can draw from that.

Below are intentionally flawed prompts. We deliberately construct poor examples to see where they break.

Example 1: “Hard or Soft” — But the Prompt Is Too Vague

The idea is simple: we want the model to determine whether an object is hard or soft.

Here are the few-shot examples:

Input: brick
Output: hard

Input: pillow
Output: soft

Input: car
Output: ???

Now notice the problems:

  • There are too few examples.

  • We abruptly change the level of ambiguity. Brick and pillow are classic, clear-cut cases. But a car is less straightforward.

  • There is a more subtle issue: “hard/soft” may be interpreted as a general characteristic, not specifically as a tactile surface property.

As a result, the model may respond unpredictably. It might classify the object along a completely different dimension — for example, as a “vehicle” or even shift toward unrelated attributes.

What Went Wrong?

Problem 1: The prefixes “Input/Output” are too generic

“Input” and “Output” do not clarify what property we are evaluating.

Is this about:

  • Material?

  • Tactile sensation?

  • Safety?

  • Category?

It is unclear.

If we instead write:

Object: brick
Surface feel: hard

Object: pillow
Surface feel: soft

The model receives more semantic guidance and less room for interpretation.

Problem 2: Insufficient Context

The model does not know we are specifically referring to physical touch or pressure, rather than a typical descriptive adjective.

Problem 3: Too Few Examples to Lock the Pattern

When examples are both minimal and obvious, the model may not reliably infer the intended rule.

Quick Fix: Restrict the Allowed Outputs

One way to improve the prompt is to add a simple rule at the top:

Your answer must be either “hard” or “soft.”

The quotation marks serve as a clear constraint: only these two labels are allowed.

Even this small clarification often helps the model understand that we are referring to physical properties. If you press on a car, it feels hard — therefore the answer is “hard.”

Example 2: Speed — Better, But Still Problematic

Now consider another few-shot set:

Object: airplane
Speed: fast

Object: worm
Speed: slow

Object: ball
Speed: ???

At first glance, this looks better. “Speed” is a measurable concept.

But the model might respond: “It depends on the situation.”

And that is logically correct.

A ball can move very fast (if kicked or launched) or slowly (if rolling gently). The model detects ambiguity.

Key Insight: Sometimes Classification Is Impossible Without Context

If we want the answer to fit “fast/slow,” we must provide enough context to remove ambiguity.

For example:

Object: ball kicked by a toddler
Speed: slow

Now the model can confidently answer: slow.

What to Remember When Designing Few-Shot Examples

1. Use meaningful prefixes.

“Input/Output” is usually weak. Prefer:

  • Object

  • Property

  • Surface feel

  • Speed

  • Sentiment

  • Category

The more semantic clarity, the better.

2. Ensure the rule is unambiguous.

If you show the examples to ten people, would they all interpret the task the same way?

If not, the model will struggle too.

3. Sometimes add a short instruction above the examples.

Ask yourself: what improves clarity with minimal prompt cost?

If a short rule makes the task explicit — add it.
If the rule is hard to describe but easy to demonstrate — rely on examples.

4. Ensure the examples themselves are unambiguous.

Even “airplane — fast” could be debated (an airplane can sit still on a runway). Sometimes that is acceptable — the model may infer the typical case.

But if you require high precision, add context directly to the examples.

5. Make sure the new input contains enough information.

You can perfectly train a pattern — and then break it by providing a new input that lacks sufficient data for classification.

Summary

Few-shot errors usually arise from three causes:

  • The format is too generic (weak prefixes, low semantic clarity)

  • There is insufficient context (unclear what is being measured)

  • The new input lacks enough information to apply the rule unambiguously

Keeping these three principles in mind significantly improves the stability and quality of few-shot prompting.
